# NSFWTrack Product Vision

本文档记录跨版本的长期产品原则。具体功能只有在独立阶段的 `GOAL.md`
明确授权后才能实现；阶段目标可以收窄能力，但不能突破本文和 `RULE.md` 的
永久安全边界。

## 产品定位

NSFWTrack 保持以下定位：

```text
NSFW-first
local-first
privacy-first
single-user
self-hosted
```

> NSFWTrack 是一个本地优先、隐私优先，以 NSFW 内容收藏、来源聚合、
> 内容获取、状态追踪和个性化发现为核心的自托管应用。

项目名称继续使用 `NSFWTrack`，不改名为 MediaTrack，也不转型为普通影视
资料库。通用数据模型可以自然兼容普通全年龄内容，但它只是附带能力，不能
主导 Provider 选择、数据模型或版本路线。项目不会为了普通影视优先建设季、
集、电视台或播出计划体系。

## 本地所有权与隐私

- 收藏、状态、评分、短评、标签、历史和推荐偏好默认只保存在本地。
- 本地数据库是用户资料的主数据源，不要求 NSFWTrack 云端账号。
- 默认不上传收藏、搜索词、标签、评分、历史或推荐偏好。
- 外部 Provider 只提供元数据、内容候选和经过批准的下载来源。
- 用户能够备份、恢复、迁移和删除自己的全部业务数据。
- 网络功能必须可见、可控，并且能够在不破坏本地资料的情况下关闭。
- 敏感查询、内容标识、凭据和原始 Provider 响应不得进入普通日志。

## 数据分层

NSFWTrack 长期管理三个相互分离的层次。

### 元数据

包括标题、别名、创作者、标签、发布日期、简介、内容类型、来源 URL、
Provider key、External ID、来源更新时间和元数据变化摘要。

### 用户记录

包括状态、评分、短评、合集、自定义标签、查看记录、编辑记录、推荐反馈和
屏蔽规则。Provider 不能成为这些数据的主数据源。

### 本地内容

包括用户导入的本地文件、封面、预览图、用户确认下载的媒体、附件、本地
媒体索引，以及来源与本地文件的关系。

外部响应、本地业务记录和本地文件必须分别存储、校验和授权。原始 Provider
响应不是本地业务数据，派生索引也不能成为写操作的授权依据。

## 受控下载

封面、预览文件和媒体下载属于长期正式能力，但默认未授权。未来独立阶段可
明确授权单项下载、用户多选后的批量下载、进度、暂停、恢复、取消、隔离
临时目录、类型和大小限制、Hash 校验、重复处理及失败回滚。

下载闭环必须遵循：

```text
展示来源和文件信息
-> 用户明确确认
-> 下载到临时区域
-> 校验
-> 移入正式目录
-> 建立本地关系
-> 刷新派生媒体索引
```

页面打开、搜索完成或后台检查不能静默开始下载。下载不得退化为任意 URL
获取器，也不得未经确认覆盖文件或执行批量写入。

## Provider 认证与秘密隔离

具体 Provider 可以在未来获批阶段声明 `none`、`api_token`、`oauth`、
`username_password` 或 `session_cookie`。认证必须来自用户自己的合法授权。

- 每个 Provider 的凭据必须隔离，Adapter 只能读取自己的凭据。
- 凭据只保存在本地；密码不得明文保存。
- 优先使用系统密钥库或经过审查的加密凭据存储。
- Cookie、Token 和密码不得进入普通 JSON 备份、普通日志或异常文本。
- UI 只显示认证状态，不能回显完整秘密。
- 认证必须支持测试、过期提示、更新和撤销。
- 不得在 Provider 间共享凭据，也不得从浏览器隐藏提取会话信息。
- 用户必须明确登录、授权或导入自己的 Cookie。

## Provider-specific 获取

未来阶段可以授权代码固定的 Provider-specific Adapter 使用官方 API、公开
JSON、登录后的结构化接口、RSS/Atom、用户导出的结构化文件，或经过单独
审查的 Provider-specific HTML 解析。

允许的是固定 Provider、固定 Host、固定 Endpoint 和固定操作；禁止的是用户
自定义 Host、协议、端口、base URL 或路径，以及整站遍历和递归发现未知链接。
具体 Provider、认证方式、解析方式和下载能力都必须由用户逐项批准。

## 本地推荐

本地推荐属于长期正式能力，但默认未授权。未来阶段可以使用喜欢、不喜欢、
忽略、评分、标签、创作者、查看历史、合集、来源偏好、屏蔽规则、新颖度和
重复度作为本地信号。

推荐应尽量解释依据，支持“不感兴趣”和按标签、创作者或来源屏蔽，并允许
关闭和清除推荐历史。推荐候选必须与正式收藏分离，不能自动加入收藏、自动
下载或默认上传本地偏好。

可选 AI 同样默认未授权；任何未来授权都必须保持本地数据最小披露，并单独
定义输入、输出、网络、日志、确认和关闭边界。

## 可控后台同步

后台同步属于长期正式能力，但默认关闭且默认未授权。未来独立阶段可以明确
授权“仅手动、每天、每周或自定义”模式，用于检查元数据变化、推荐候选、
下载状态、失效来源、Provider 认证状态或附件信息。

后台活动必须可见、可控、可关闭、可撤销，并清楚展示最近运行、来源、结果
和错误。默认更新流程是“发现变化 -> 通知 -> 展示差异 -> 用户确认 -> 写入
本地数据库”。任何隐藏网络或未经确认覆盖用户编辑内容的行为永久禁止。

## 永久禁止项

以下能力不能由任何 `GOAL.md` 开放：

- 任意 URL 获取器，或用户自定义 Host、协议、端口、base URL 和任意路径；
- 无限制通用爬虫、整站遍历、递归发现或抓取未知链接；
- 绕过登录、年龄验证、付费、订阅、地区、账号权限或其他访问控制；
- 伪造授权、利用漏洞或访问用户无权访问的第三方内容；
- 窃取账号、密码、Cookie、Token 或浏览器会话；
- 隐藏提取、泄漏、跨 Provider 共享或越权读取凭据；
- 隐藏外部请求或后台网络活动；
- 未经确认的大量写入、覆盖或下载；
- 默认上传本地收藏、用户记录或推荐偏好；
- 未展示差异就自动覆盖用户编辑内容。

## 搜索、预览、写入与下载分离

外部搜索、详情预览、本地写入和下载是四种独立行为：

```text
用户主动搜索
-> 展示 Provider 结果
-> 用户选择结果
-> 获取并展示详情
-> 展示将新增或修改的字段
-> 用户选择保存方式
-> 用户明确确认
-> 写入数据库或开始独立下载
```

长期可以支持单条确认、多条选择后统一确认、只保存元数据、保存并下载、
创建新 Item、关联已有 Item、稍后处理和逐字段覆盖规则。无论采用哪种模式，
都必须做到来源明确、结果可预览、行为由用户授权。
