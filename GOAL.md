# GOAL.md

# 当前目标：Phase 2-I1 性能基线与数据库查询审查

请先读取 `RULE.md`、`PLAN.md`。
长期边界以 `RULE.md` 为准。

---

## 一、目标

建立可重复的本地性能基线，找出：

- N+1 查询
- 重复查询
- 无界查询
- 缺少分页的列表
- 不必要的全表扫描
- 统计页和工作台的慢查询
- 大数据量下可能退化的操作

本轮只分析、测试和记录问题，不修改数据库结构。

---

## 二、范围

重点审查：

- 条目列表筛选、排序、分页
- 首页工作台
- 统计页
- 标签、创作者、合集页面
- saved views 与 activity
- 重复条目与元数据清理
- 数据健康检查
- 备份校验与导入 dry-run

本轮不做：

- 新增索引
- 修改表或字段
- 编写真实迁移
- 缓存系统
- 后台任务
- 外部监控
- 前端框架重构

---

## 三、测试数据与安全

所有性能验证使用临时数据库或隔离数据卷。

要求：

- 不使用或修改默认 schema 2 数据卷
- 不修改真实业务数据
- 可生成小型、中型、大型测试数据集
- 测试结束后清理临时数据
- 不把生成数据提交到仓库

建议规模：

- 100 条
- 1,000 条
- 10,000 条

---

## 四、性能审查

建议新增：

- `app/services/performance_audit.py`
- `scripts/profile_queries.py`
- `tests/test_performance_audit.py`
- `PERFORMANCE.md`

至少记录：

- 页面或操作名称
- 数据规模
- SQL 查询数量
- 执行时间
- `EXPLAIN QUERY PLAN` 摘要
- 是否存在 N+1
- 是否存在全表扫描
- 风险等级
- 后续优化建议

要求：

- 工具默认只读
- 不执行任意用户 SQL
- 不依赖外部服务
- 不新增第三方依赖

---

## 五、重点判定

必须检查：

- 列表页是否保持分页
- 模板循环中是否触发额外 SQL
- 标签 / 创作者 / 合集计数是否逐行查询
- 工作台最近数据和 saved views 是否有重复查询
- 统计页是否重复扫描相同数据
- duplicates / cleanup 是否随数据量明显恶化
- data-health 是否存在无界明细加载
- 导入与备份 preview 是否把大文件全部重复解析

本轮只记录结论，不直接优化。

---

## 六、输出

`PERFORMANCE.md` 至少包含：

- 当前性能基线
- 各页面查询数量与耗时
- 已确认的性能问题
- 暂未发现问题的路径
- 优先级排序
- 建议在 I2 实施的索引或查询优化
- 哪些优化需要真实 schema 迁移

不得把猜测写成已确认问题。

---

## 七、测试与文档

要求：

- 性能审查工具自身有测试
- 不写依赖固定毫秒数的脆弱测试
- 可测试查询数量上限和是否出现 N+1
- 全量回归测试通过
- Docker build / up / `/login` 200 / down 通过
- 更新 README、TASKS、REVIEW、CHANGELOG、PLAN
- CHANGELOG 只写 `Unreleased`
- 不修改旧 tag，不创建 Release

---

6. 性能问题优先级
7. 是否修改数据库结构或业务逻辑
