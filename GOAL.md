# GOAL.md

# NSFWTrack 当前开发目标

本轮开发目标：
在保持 Phase 1 本地 MVP 边界不变的前提下，增强“备份恢复”功能的安全性、可用性和用户提示。

本轮重点是：

* 限制上传备份文件大小
* 增加恢复前预览
* 明确“合并恢复，不是覆盖恢复”
* 优化错误提示
* 增加对应测试

不新增外部源、不新增复杂业务功能。

---

## 一、必须遵守的项目边界

NSFWTrack 是项目名称。

Phase 1 严格限定为：

> 本地单用户内容记录器 / 收藏管理器 MVP。

本轮只允许实现：

* 本地 JSON 备份文件大小限制
* 本地 JSON 备份恢复前预览
* 本地 JSON 备份恢复错误提示优化
* 备份页面文案增强
* 相关测试
* README / TASKS / REVIEW 文档更新

本轮禁止实现：

* 外部内容源
* 爬虫
* adapter
* 远程图片拉取
* cookie / token 管理，登录 session 和语言偏好 session 除外
* 自动同步
* 多源搜索
* 随机探索接口
* 推荐系统
* AI 助手
* 从 URL 导入备份
* 云端备份
* 定时备份

如果开发中发现上述超范围内容，不要实现，直接跳过并说明原因。

---

## 二、本轮任务 1：上传文件大小限制

请为 JSON 备份上传增加文件大小限制。

要求：

1. 只限制备份恢复上传文件。
2. 默认最大上传大小建议为 `5MB`。
3. 可以通过环境变量配置，例如：

```env
MAX_BACKUP_UPLOAD_MB=5
```

4. 如果未配置，则默认 `5MB`。
5. 如果上传文件超过限制，应返回清晰错误提示。
6. 错误提示需要支持中文 / English。
7. 超限文件不能进入恢复逻辑。
8. 不要影响 JSON / CSV 导出功能。

---

## 三、本轮任务 2：恢复前预览

请新增 JSON 备份恢复前预览功能。

要求：

1. 只允许上传本地 JSON 文件。
2. 预览接口必须要求登录。
3. 预览时只解析并校验文件，不写入数据库。
4. 预览内容至少包含：

   * items 数量
   * tags 数量
   * creators 数量
   * item_tags 数量
   * item_creators 数量
   * user_item_states 数量
   * backup schema
5. 如果 JSON 格式错误，应返回清晰错误提示。
6. 如果 schema 不匹配，应返回清晰错误提示。
7. 如果文件过大，应返回清晰错误提示。
8. 预览功能不得修改数据库。

建议接口：

```text
POST /api/backup/preview/json
```

请求方式可以继续使用文件上传。

---

## 四、本轮任务 3：恢复页面体验增强

请优化 `/backup` 页面。

要求：

1. 页面明确说明：

> JSON 恢复采用“合并恢复”策略，不会清空当前数据库。

2. 页面明确说明：

> 恢复只支持由 NSFWTrack 导出的 JSON 备份文件。

3. 页面明确说明：

> 不支持 URL 导入、云端同步或外部数据源。

4. 页面增加“预览备份”操作。
5. 用户可以先预览，再恢复。
6. 第一版可以采用简单表单实现，不需要复杂前端。
7. 继续保持 FastAPI + Jinja2 + HTMX。
8. 中文 / English 文案都需要接入 i18n。

---

## 五、本轮任务 4：错误提示优化

请优化备份相关错误提示。

至少覆盖：

1. 未上传文件。
2. 文件名不是 `.json`。
3. 文件大小超过限制。
4. JSON 格式错误。
5. backup schema 不匹配。
6. 备份内容缺少必要字段。
7. 恢复过程中发生异常。

要求：

* 错误提示需要用户能看懂。
* 中文 / English 都要支持。
* 不要在页面或日志中输出敏感信息。
* 不要输出完整异常堆栈到用户页面。

---

## 六、本轮任务 5：测试

请新增或更新测试，至少覆盖：

1. 未登录访问 preview 接口应失败。
2. 登录后上传合法 JSON 可以预览成功。
3. 预览合法 JSON 不会修改数据库。
4. 上传非法 JSON 预览失败。
5. 上传非 `.json` 文件失败。
6. 上传超过大小限制的文件失败。
7. schema 不匹配时预览失败。
8. 合法 JSON 仍然可以恢复成功。
9. 恢复失败时现有数据库不被破坏。
10. 中文 / English 备份页面均能渲染新的提示文案。
