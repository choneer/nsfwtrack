# GOAL.md

# 当前目标：Phase 4-M4 — 媒体写入协调与索引自动一致性

## 目标

在 Phase 4-M3 签名媒体索引之上，
统一协调应用内所有媒体文件写操作与索引刷新。

正常完成的媒体写操作应自动产生一份新的完整索引快照；
结果不确定或刷新失败时必须安全失效旧索引，
不得让只读页面继续把旧快照显示为有效。

## 1. 跨进程媒体操作锁

新增统一媒体操作锁，用于串行化：

- 媒体上传
- 单文件 rename / move
- 批量 rename / move
- alias keeper 归一化
- duplicate cleanup
- damaged cleanup
- recovery restore
- cleanup anchor 删除
- 媒体根初始化
- 手动增量刷新
- 手动完整验证 / 重建

要求：

- 使用固定的应用数据目录锁文件，不放入媒体根和仓库
- 锁路径不得由请求参数控制
- 锁文件必须是普通文件，权限限制为当前应用用户
- 拒绝 symlink、目录和特殊对象
- 支持跨进程互斥，不只使用线程锁
- 获取锁具有合理超时
- 获取失败必须在文件或业务数据变更前返回 `media_busy`
- GET 页面不得获取写锁或产生文件
- 锁释放必须覆盖成功、异常、重定向和客户端中断路径

锁只协调 NSFWTrack 自身写操作，
不宣称能够阻止用户在应用外直接修改媒体目录。

## 2. 写后索引刷新

已确认成功并实际改变媒体文件系统的操作完成后：

- 在业务操作及其数据库事务完成后执行一次增量索引刷新
- 刷新仍使用 M3 的安全 FD 扫描和完整快照原子替换
- 不允许在业务数据库事务内部扫描整个媒体目录
- 不允许使用索引作为原文件操作的授权证据
- 写操作与刷新必须处于同一媒体操作锁生命周期内

覆盖操作：

- 上传新增文件
- rename / move 路径变化
- 批量 rename / move
- alias 归一化后的路径删除
- duplicate cleanup
- damaged cleanup
- recovery 创建普通媒体
- cleanup anchor 删除
- 媒体根初始化

批量操作无论包含多少成功项，每个请求最多刷新一次。

以下纯数据库操作不刷新媒体索引：

- 设置、替换或清除 cover/avatar 引用
- 条目、标签、创作者、合集和状态修改
- 仅修改引用而不改变任何媒体路径的操作

## 3. 结果分类

为媒体变更结果增加统一索引处理分类：

### no_filesystem_change

- 没有媒体文件或目录变化
- 不刷新，不失效索引

### filesystem_changed_known

- 最终文件系统状态明确
- 执行一次增量刷新

### filesystem_changed_partial_known

- 部分操作成功，部分失败
- 最终实际路径集合明确
- 在全部单项完成后执行一次增量刷新

### filesystem_outcome_unknown

适用于：

- commit outcome unknown
- 无法确定源或目标路径归属
- 独立复核失败
- 文件删除结果不明确
- 清理回滚结果不明确

处理要求：

- 禁止尝试基于推测更新索引
- 将索引标记为 invalid
- 使用明确 stale reason
- 保留原业务操作的实际警告和结果
- 不把操作报告为完全成功

## 4. 刷新失败处理

业务操作成功，但后续索引刷新失败时：

- 不回滚已经完成的文件或业务数据库变更
- 不把原业务操作改报为失败
- 将索引标记为 invalid
- stale reason 使用 `post_mutation_refresh_failed`
- 向用户显示“操作已完成，但索引需要手动刷新”
- 不泄漏主机绝对路径、异常对象或凭据
- 之后只读页面安全降级到文件系统扫描

若连索引失效状态都无法写入：

- 保留原业务结果
- 显示更高等级警告
- 不伪称索引已同步

## 5. 扫描中心协调

手动增量刷新和完整验证必须使用相同媒体操作锁。

当另一个媒体操作正在执行时：

- 扫描请求不得并行进入
- 超时后返回 `media_busy`
- 不产生半套索引
- 不影响正在执行的媒体操作

当扫描中心持有锁时：

- 新媒体写操作等待或安全拒绝
- 不允许扫描快照与应用写入交叉提交

## 6. UI 与状态

成功媒体操作应显示：

- 业务操作结果
- 索引已同步，或
- 操作完成但索引已失效，需要手动刷新

扫描中心显示：

- 最近索引刷新来源：
  - manual_incremental
  - manual_full
  - post_upload
  - post_rename
  - post_move
  - post_batch
  - post_cleanup
  - post_recovery
  - post_root_init
- 最近刷新失败原因
- 当前是否因未知媒体操作而失效

保留“索引是时间点快照，应用外文件变更可能使其过期”的提示。

## 7. 安全与范围边界

- 不新增后台线程、队列、计划任务或文件监听器
- 不在 GET 中自动刷新
- 不自动周期扫描
- 不新增网络、爬虫、远程媒体、识别、推荐或 AI
- 不扩大任何现有删除、移动或恢复权限
- 不修改媒体操作现有确认规则
- 不修改备份格式
- 不新增依赖
- 不新增数据库表或迁移
- 应用版本保持 1.0.6
- Schema 保持 3
- 不创建 tag、Release，不部署 N100

## 8. 测试要求

必须覆盖：

- 两个并发媒体写操作不会交叉执行
- 手动扫描与媒体写操作不会并行提交
- 锁超时在任何变更前安全失败
- 锁文件 symlink / 目录 / 特殊对象安全拒绝
- 上传成功后索引自动包含新文件
- rename / move 后旧路径消失、新路径出现
- batch 多项成功只刷新一次
- batch 部分成功后索引与实际最终状态一致
- duplicate / damaged cleanup 后索引移除实际删除路径
- recovery 后索引包含恢复媒体
- alias 归一化后索引反映 keeper 和残留路径
- commit unknown 导致索引失效，不执行推测刷新
- 写后刷新失败不回滚已完成业务操作
- 刷新失败后页面降级到 filesystem
- 纯引用修改不触发媒体扫描
- GET 继续保持数据库与文件系统零写入
- Docker 重建后锁文件和有效索引行为正常

## 完成标准

- 所有应用内媒体写操作均纳入统一锁和索引结果分类
- 已知成功操作完成后只刷新一次
- 未知结果绝不保留“有效”的旧索引
- 刷新失败不破坏业务操作结果
- 只读页面在正常写操作后继续直接使用 index
- 应用外变更仍通过时间点过期提示和手动刷新处理
- 中英文、CHANGELOG、TASKS、REVIEW、PLAN 完整
- 专项、并发、核心组合、全量 pytest、pip check、Docker 和 Actions 通过
