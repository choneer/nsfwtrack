# 当前目标：Phase 3-B3 — keeper 删除竞态修复

当前稳定版与最新 Release 为 `v1.0.6`，Phase 3-B1 / B2 已正式发布。
B3 仅记录在 CHANGELOG Unreleased，应用版本保持 1.0.6，Schema 保持 2。

## 目标

关闭引用提交后、冗余文件删除期间 keeper 被删除或替换时可能丢失
最后有效副本并产生断裂引用的竞态，同时保持既有 B3 功能和边界。

## 任务

- 删除前从已验证 keeper 创建同文件系统随机安全锚点并持久化文件与目录
- 删除窗口内让所有相关封面 / 头像引用指向存在、合法、完整 SHA-256 正确的锚点
- keeper 在首删前、中途或最后一次删除期间缺失时，从锚点无覆盖恢复
- keeper 路径被替换或占用时不覆盖外部文件，发布唯一恢复路径并迁移引用
- 成功时清理安全锚点并 fsync；失败时明确报告且至少保留一份有效同哈希媒体
- 增加三个删除时序、引用持续有效、外部文件不覆盖及临时文件清理测试
- 保持原 B3 显式 keeper、只读预览、重扫、危险确认、删除失败重试和 A3/A4 边界
- 同步中文 / English、测试、CHANGELOG Unreleased 与当前开发文档

## 边界

- 不自动选择保留文件
- 每次只处理一个当前重复组
- 不接受损坏、符号链接、越界、缺失或非组成员路径
- 不删除用户选择的保留文件
- 不覆盖 keeper 路径上外部新出现的文件
- 不修改 A3/A4 候选算法
- 不请求外部网络，不使用 AI 或图像识别
- 不新增表，不修改 Schema 2、迁移、依赖或版本
- 不创建 tag 或 Release，不部署到 N100

## 完成标准

- 删除任何冗余文件时均存在一份已验证安全锚点
- 数据库引用在首删前、中途和末次删除竞态中始终解析到合法同哈希文件
- keeper 缺失时恢复有效文件，路径被占用时迁移到唯一恢复路径
- 成功、异常和重试后均无临时锚点残留
- 外部替换文件、非目标文件、其他媒体组和 A3/A4 保持不变
- pytest、pip check、Docker 和 Actions 通过
