# GOAL.md

# NSFWTrack 当前开发目标

本轮开发目标：  
进入 Phase 2-A2，在保持项目本地单用户边界不变的前提下，为条目列表页增加批量编辑能力。

本轮重点是：

- 条目多选
- 批量修改状态
- 批量添加标签
- 批量移除标签
- 批量设置评分
- 批量删除
- 删除二次确认
- 批量操作结果提示
- 中英文文案覆盖
- 相关测试

本轮不接入外部内容源，不做爬虫，不做推荐系统，不做 AI 助手。

---

## 一、必须遵守的项目边界

NSFWTrack 是项目名称。

Phase 2-A2 的目标是：

> 在 Phase 2-A1 高级筛选 / 排序 / 分页的基础上，让用户能对筛选出来的本地条目进行批量管理。

本轮只允许实现：

- 本地条目多选
- 本地条目批量状态修改
- 本地条目批量添加标签
- 本地条目批量移除标签
- 本地条目批量评分
- 本地条目批量删除
- 批量删除二次确认
- 批量操作成功 / 失败提示
- 中英文 i18n 文案
- 相关测试
- README / TASKS / REVIEW / CHANGELOG 文档更新

本轮禁止实现：

- 外部内容源
- 爬虫
- adapter
- 远程图片拉取
- 自动同步
- 多源搜索
- 随机探索接口
- 推荐系统
- AI 助手
- URL 导入
- 云端同步
- 云端备份
- 定时任务
- 多用户系统
- 复杂权限系统
- 大型前端框架
- 大范围重构
- 覆盖式备份恢复
- 修改 v0.1.0 tag
- 创建新的 GitHub Release

如果开发中发现上述超范围内容，不要实现，直接跳过并说明原因。

## 二、本轮任务 1：列表页条目多选

请在条目列表页增加多选能力。

要求：

1. 每个条目卡片或列表项支持选择。
2. 页面提供“全选当前页”功能。
3. 页面提供“取消选择”功能。
4. 多选状态只针对当前页面展示结果即可。
5. 第一版不需要跨分页选择全部结果。
6. 不要引入复杂前端状态管理。
7. 继续保持 FastAPI + Jinja2 + HTMX。
8. 不引入 React / Vue。
9. 未登录访问仍应跳转登录页。
10. 中文 / English 文案都需要覆盖。

说明：

- 当前页全选即可。
- 不要求“选择所有筛选结果”。
- 不要求跨页批量操作。
- 不要求持久化选择状态。

## 三、本轮任务 2：批量修改状态

请实现批量修改条目状态。

支持状态：

- 想看
- 在看
- 已看
- 喜欢
- 不喜欢
- 忽略

要求：

1. 用户选择多个条目后，可以一次性修改状态。
2. 后端仍使用现有内部状态枚举。
3. API 字段名不要翻译。
4. 页面展示文案需要中文 / English。
5. 批量操作必须要求登录。
6. 没有选择条目时，应给出清晰错误提示。
7. 状态值非法时，应安全拒绝，不要 500。
8. 操作完成后应返回列表页，并显示成功提示。
9. 操作后尽量保留当前筛选 / 排序 / 分页参数。

---

## 四、本轮任务 3：批量添加标签

请实现批量添加标签。

要求：

1. 用户选择多个条目后，可以选择一个已有标签进行批量添加。
2. 如果条目已经有该标签，不应重复创建关联。
3. 标签不存在时，应安全拒绝，不要自动创建新标签。
4. 没有选择条目时，应给出清晰错误提示。
5. 没有选择标签时，应给出清晰错误提示。
6. 操作完成后显示成功提示。
7. 操作后尽量保留当前筛选 / 排序 / 分页参数。
8. 中文 / English 文案都需要覆盖。

第一版只支持添加一个已有标签，不做批量创建新标签。

---

请实现批量移除标签。

要求：

3. 标签不存在时，应安全拒绝。
4. 没有选择条目时，应给出清晰错误提示。
5. 没有选择标签时，应给出清晰错误提示。
6. 操作完成后显示成功提示。
7. 操作后尽量保留当前筛选 / 排序 / 分页参数。
8. 中文 / English 文案都需要覆盖。

## 六、本轮任务 5：批量设置评分

请实现批量设置评分。

要求：

1. 用户选择多个条目后，可以批量设置评分。
2. 支持评分范围建议为：
   - 1
   - 2
   - 3
   - 4
   - 5

4. 评分值非法时，应安全拒绝，不要 500。
5. 没有选择条目时，应给出清晰错误提示。
6. 操作完成后显示成功提示。
7. 操作后尽量保留当前筛选 / 排序 / 分页参数。
8. 中文 / English 文案都需要覆盖。

---

## 七、本轮任务 6：批量删除

请实现批量删除条目。

要求：

1. 用户选择多个条目后，可以批量删除。
2. 删除前必须有二次确认。
3. 第一版可以使用浏览器原生 confirm。
4. 没有选择条目时，应给出清晰错误提示。
5. 删除后应清理相关关联数据：
   - item_tags
   - item_creators
   - user_item_states
6. 删除操作必须要求登录。
7. 删除完成后显示成功提示。
8. 操作后返回列表页。
9. 不要误删未选择条目。
10. 中文 / English 文案都需要覆盖。

注意：

- 批量删除是危险操作，必须有明显提示。
- 不做回收站。
- 不做软删除，除非当前项目已有软删除机制。
- 不做覆盖式恢复。

---

## 八、本轮任务 7：后端批量操作整理

请整理批量操作后端逻辑。

建议新增 service：

- `app/services/bulk_actions.py`

也可以在现有 service 中扩展，但不要让 router 变得过长。

要求：

1. 批量操作逻辑清晰可维护。
2. 对条目 ID 做基础校验。
3. 只操作本地 SQLite 数据。
4. 不拼接不安全 SQL。
5. 优先使用 SQLAlchemy ORM。
6. 对不存在的条目 ID 应安全跳过或返回清晰结果。
7. 批量操作应尽量使用事务。
8. 操作结果应返回：
   - 处理了多少条
   - 跳过了多少条
   - 是否成功
9. 不要引入外部任务队列。
10. 不要引入后台异步任务。

---

## 九、本轮任务 8：页面 UI 与交互

请优化列表页批量操作 UI。

要求：

1. 列表页顶部或筛选区域下方增加“批量操作”区域。
2. 批量操作区域包含：
   - 当前页全选
   - 取消选择
   - 批量状态选择
   - 批量添加标签
   - 批量移除标签
   - 批量评分
   - 批量删除
3. 批量删除按钮需要明显危险样式。
4. 批量删除需要二次确认。
5. 没有条目时，不应显示混乱的批量操作。
6. 没有选择条目时，应给出明确提示。
7. 操作成功后显示统一 flash message。
8. 操作失败后显示统一 error message。
9. 页面文案支持中文 / English。
10. UI 保持轻量，不引入大型前端框架。

可以使用少量原生 JavaScript 实现：

- 全选当前页
- 取消选择
- 删除确认

但不要引入复杂前端构建流程。

## 十、本轮任务 9：保留筛选参数

批量操作完成后，应尽量返回用户原来的列表状态。

要求：

1. 保留当前搜索关键词。
2. 保留当前状态筛选。
3. 保留当前标签筛选。
4. 保留当前创作者筛选。
5. 保留当前最低评分筛选。
6. 保留当前时间范围。
7. 保留当前排序。
8. 保留当前 page_size。
9. page 可以回到当前页，若当前页因删除后不存在，应安全回退。
10. 不要因为 query string 异常导致 500。

建议：

- 使用隐藏字段保存当前 query string。
- 或使用 `next` 参数返回原列表 URL。
- `next` 必须做安全校验，只允许站内相对路径。
- 不允许开放重定向。
