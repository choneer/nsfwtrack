# NSFWTrack 当前开发目标

本轮开发目标：  
进入 Phase 2-D1，在保持项目本地单用户边界不变的前提下，新增“重复条目检测与手动合并”能力，帮助用户处理导入、整理过程中产生的重复条目。

本轮重点：

- 重复条目候选检测

本轮只做本地重复条目检测与手动合并。

## 二、本轮任务 1：重复检测规则

请实现本地重复候选检测。

第一版只做安全、可解释的规则，不做复杂模糊匹配。

至少支持两类检测：

### 1. 标题完全匹配

规则：

```text
title 完全相同
```

要求：

- 忽略首尾空格。
- 不改变原始标题。
- 只用于检测候选。
- 不自动合并。

### 2. 标题归一化匹配

建议归一化规则：

```text
Unicode NFKC
去除首尾空格
casefold / lower
连续空白折叠为单个空格
```

可选增强：

```text
中英文常见全角/半角差异归一化
```

要求：

- 归一化只用于检测候选。
- 不修改数据库中的原始标题。
- 不做外部搜索。
- 不做 AI 判断。
- 不做图片相似度。
- 不做复杂模糊算法。
- 不新增依赖。
- 检测结果必须标明匹配类型：
  - exact_title
  - normalized_title

如果两个条目只是疑似重复，必须由用户手动确认合并。

## 五、本轮任务 4：合并策略

请实现两个条目的手动合并。

定义：

```text
primary item = 保留条目
duplicate item = 被合并后删除的条目
```

合并后：

- primary item 保留。
- duplicate item 删除。
- duplicate 的关联数据合并到 primary。
- duplicate 本身被删除。
- 合并过程应尽量事务安全。

### 必须合并的关联数据

请合并：

1. 标签关系
2. 创作者关系
3. 合集关系

要求：

- primary 缺少的关系从 duplicate 复制过来。
- primary 已有的关系不要重复创建。
- 合并后不要产生重复关联。
- 删除 duplicate 后，关联关系应正确清理。
- 不要删除标签。
- 不要删除创作者。
- 不要删除合集。

### 状态 / 评分 / 短评合并

请支持安全合并 `UserItemState`。

第一版策略建议：

1. 如果 primary 没有状态记录，duplicate 有状态记录：
   - 复制 duplicate 的状态记录到 primary。

2. 如果 primary 有状态记录，duplicate 没有：
   - 保留 primary。

3. 如果两边都有状态记录：
   - 默认保留 primary。
   - 页面展示冲突。
   - 用户可以选择是否用 duplicate 覆盖 primary 的：
     - status
     - rating
     - note

要求：

- 不要无提示覆盖 primary 数据。
- 空值不要错误覆盖非空值，除非用户明确选择。
- 非法状态 / 非法评分不能写入。
- 合并后 duplicate 的状态记录随 duplicate 删除清理。
- 合并成功后显示摘要。

### summary 合并

建议策略：

1. primary summary 为空，duplicate summary 非空：
   - 自动复制 duplicate summary 到 primary。

2. primary summary 非空，duplicate summary 也非空且不同：
   - 默认保留 primary。
   - 页面提示存在冲突。
   - 用户可以选择是否使用 duplicate summary 覆盖。

要求：

- 不要无提示覆盖已有 summary。
- 不要丢失用户明确选择保留的数据。

### extra JSON 合并

建议策略：

1. primary extra 为空，duplicate extra 非空：
   - 复制 duplicate extra 到 primary。

2. primary extra 非空，duplicate extra 为空：
   - 保留 primary。

3. 两边都有 extra：
   - 浅层合并不冲突 key。
   - 冲突 key 默认保留 primary。
   - 冲突信息在页面或结果摘要中提示。

要求：

- 不要因为 extra JSON 结构异常导致 500。
- 不要盲目覆盖 primary 的 extra。
- 不要丢失 duplicate 中非冲突 key。
- 合并结果必须仍然是合法 JSON 结构。
