# 当前目标：Phase 4-M2 — 批量整理与别名归一化

## 目标

在 Phase 4-M1 单文件安全能力之上，
增加普通媒体的批量移动、批量重命名和硬链接别名手动归一化。

## 功能范围

### 1. 多选与批量预览

- 媒体库和目录页支持选择多个有效普通媒体
- 别名页支持选择单个别名组并指定 keeper
- 所有 GET 预览保持数据库和文件系统零写入
- 服务端重新扫描并验证每个路径，不信任前端提供的文件事实
- 展示完整路径、SHA、身份、引用、目标和预计结果
- 限制单批数量，拒绝重复、非法、缺失和不合格路径

### 2. 批量安全移动

- 仅移动到媒体根内已存在的普通目录
- 默认保留原 basename，可在预览中逐项修改
- 禁止覆盖、跨根、symlink 目录、anchor、residue、damaged 和 skip
- 每个文件复用 M1 的已验证 source/target 目录 FD 链
- 精确迁移全部 item cover 和 creator avatar 引用
- 每个文件独立事务、独立结果；单项失败不回滚已完成的其他项
- commit 不确定时保留安全路径并停止该项后续删除

### 3. 批量安全重命名

- 仅允许同目录修改 basename
- 严格保持每个文件原扩展名及大小写
- 执行前检查空名称、保留前缀、长度、重复目标和已有对象
- 禁止名称交换、循环映射和临时中间名称
- 不支持正则、通配符、脚本或自动生成名称

### 4. 硬链接别名归一化

- 只接受按相同 dev/inode 验证的真实别名组
- 用户明确指定一个 keeper 路径
- 预览全部路径及 item/creator 引用分布
- 事务内将其他 alias 路径的全部引用迁移到 keeper
- commit 后逐项身份绑定删除零引用冗余 alias 路径
- 删除失败时 keeper 和全部引用保持有效，并准确报告残留路径
- commit 结果未知时不删除任何 alias 路径
- 相同 SHA、不同 dev/inode 的独立重复文件不参与归一化

## 边界

- 不创建、删除或重命名目录
- 不自动选择 keeper
- 不批量清除封面或头像引用
- 不删除独立媒体文件
- 不保存批量任务历史，不新增数据库表
- 不修改版本 1.0.6、Schema 2、迁移、依赖或 Docker/CI
- 不请求网络，不增加爬虫、识别、推荐或 AI
- 不创建 tag、Release，不部署 N100

## 完成标准

- 批量预览完全零写入
- 每项结果明确区分成功、失败、已提交但源保留和结果未知
- 任意失败不会产生断裂引用或错误删除外部对象
- 目标冲突、目录替换、身份变化和引用变化均安全拒绝
- 别名归一化后全部引用精确指向 keeper
- 独立重复文件与 alias 始终严格区分
- 中英文、测试和 Unreleased 文档完整
- pytest、pip check、Docker 和 Actions 通过
