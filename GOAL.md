# GOAL.md

# 当前目标：Phase 2-H1 数据库版本记录与升级预检

请先读取 `RULE.md`、`PLAN.md`，再执行本文件。
长期边界以 `RULE.md` 为准。

---

## 一、目标

建立数据库结构版本记录和升级预检机制。

本轮实现：

- 数据库 schema 版本记录
- 启动时版本检查
- 旧数据库安全登记基线版本
- 数据库版本状态展示
- 版本不兼容时给出清楚错误和备份提示

本轮不做：

- 修改已有业务表字段
- 删除表或字段
- 自动执行真实迁移
- 自动降级
- 自动恢复备份
- Alembic
- 外部服务

---

## 二、数据库设计

本轮允许新增一个内部表：

- `schema_migrations`

建议字段：

- `version`
- `name`
- `applied_at`

要求：

- `version` 唯一
- 当前版本使用代码常量，例如 `CURRENT_SCHEMA_VERSION = 1`
- 不修改任何现有表
- 不新增依赖
- 不把 `schema_migrations` 当作用户业务数据
- JSON 备份不得导出或恢复该内部表

---

## 三、版本初始化

启动时处理：

1. 新数据库：
   - 正常创建现有表
   - 登记当前基线版本

2. 旧数据库无 `schema_migrations`：
   - 先检查必要表是否存在
   - 结构符合当前预期时登记基线版本
   - 结构异常时不得盲目登记，必须报告错误

3. 已有版本记录：
   - 等于当前版本：正常运行
   - 高于当前版本：拒绝继续并提示应用版本过旧
   - 低于当前版本：提示需要升级，不自动迁移

要求：

- 不清空数据库
- 不重建业务表
- 不修改业务数据
- 初始化失败不能留下错误版本记录
- 异常信息不得包含完整堆栈或敏感路径

---

## 四、状态展示

在设置页增加数据库状态区域，或新增只读页面：

- 当前应用 schema 版本
- 当前数据库 schema 版本
- 状态：正常 / 需要升级 / 应用过旧 / 无法确认
- 最近登记时间
- 升级前备份提示

要求：

- 需要登录
- 页面只读
- 不提供手动修改版本按钮
- 不提供降级按钮
- 不提供跳过检查按钮

---

## 五、安全边界

必须保证：

- 用户不能提交任意版本号
- 用户不能通过 URL 修改 schema 版本
- `schema_migrations` 不进入备份恢复
- 旧备份恢复不能覆盖本地 schema 版本
- 版本检查失败时不执行后续迁移
- 不新增自动修复或强制升级
- 不修改已发布 tag

---

## 六、测试

至少覆盖：

- 新数据库登记当前版本
- 旧数据库无版本表时安全登记基线
- 必要表缺失时不盲目登记
- 当前版本匹配时正常启动
- 数据库版本高于应用时拒绝
- 数据库版本低于应用时提示升级
- 初始化失败不留下错误记录
- 状态页需要登录且只读
- 用户无法修改版本
- JSON 备份不包含 `schema_migrations`
- 旧备份恢复不覆盖 schema 版本
- i18n 和全量回归测试通过
