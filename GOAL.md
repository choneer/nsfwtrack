# GOAL.md

# NSFWTrack 当前开发目标

本轮开发目标：  
进入 Phase 2-A3，在保持项目本地单用户边界不变的前提下，增强条目详情页的展示、编辑和返回体验。

本轮重点是：

- 条目详情页信息分区
- 更清楚展示标签、创作者、状态、评分、短评、extra JSON
- 详情页快速修改状态 / 评分 / 短评
- 详情页添加 / 移除标签
- 详情页关联 / 解除创作者
- 从列表进入详情后返回时保留筛选 / 排序 / 分页状态
- 中英文文案覆盖
- 相关测试

本轮不接入外部内容源，不做爬虫，不做推荐系统，不做 AI 助手。

## 一、必须遵守的项目边界

NSFWTrack 是项目名称。

Phase 2-A3 的目标是：

本轮只允许实现：

- 本地条目详情页展示增强

## 二、本轮任务 1：详情页信息分区

请增强条目详情页展示。

详情页至少分为以下信息区域：

1. 基本信息
   - 标题
   - 描述
   - 创建时间
   - 更新时间
   - extra JSON

2. 状态信息
   - 当前状态
   - 评分
   - 短评 / 备注

3. 标签信息
   - 当前已关联标签
   - 添加标签入口
   - 移除标签入口

4. 创作者信息
   - 当前已关联创作者
   - 添加创作者入口
   - 移除创作者入口

5. 操作区域
   - 编辑条目
   - 删除条目
   - 返回列表

要求：

- 页面结构清晰，适合后续继续扩展。
- 不引入 React / Vue。
- 继续保持 FastAPI + Jinja2 + HTMX。
- 中文 / English 文案都需要覆盖。
- 项目名 NSFWTrack 不要翻译。
- API 字段名不要翻译。

## 三、本轮任务 2：详情页状态 / 评分 / 短评编辑

请在详情页支持快速修改条目的状态信息。

支持编辑：

1. 状态
   - 想看
   - 在看
   - 已看
   - 喜欢
   - 不喜欢
   - 忽略

2. 评分
   - 1
   - 2
   - 3
   - 4
   - 5
   - 可选：清空评分

3. 短评 / 备注
   - 普通文本
   - 第一版不需要富文本

要求：

- 使用现有内部状态枚举。
- 不要翻译 API 字段名。
- 状态值非法时应安全拒绝，不要 500。
- 评分非法时应安全拒绝，不要 500。
- 空短评应允许保存。
- 如果条目还没有 `UserItemState`，保存时应安全创建。
- 如果条目已有 `UserItemState`，保存时应安全更新。
- 操作必须要求登录。
- 操作成功后显示 flash success。
- 操作失败后显示 flash error。
- 操作后返回当前详情页。
- 如从列表页带有 `next` 参数进入详情，保存后仍应尽量保留返回列表的上下文。

## 四、本轮任务 3：详情页标签管理

请在详情页支持添加和移除标签关联。

### 添加标签

要求：

1. 只能从已有标签中选择。
2. 不在详情页自动创建新标签。
3. 标签不存在时安全失败，不要 500。
4. 已有关联时不要重复创建。
5. 操作成功后显示成功提示。
6. 操作失败后显示错误提示。
7. 操作后返回当前详情页。
8. 中文 / English 文案都需要覆盖。

### 移除标签

要求：

1. 可以从当前已关联标签中移除。
2. 如果条目没有该标签，应安全跳过或显示合理提示。
3. 标签不存在时安全失败。
4. 操作成功后显示成功提示。
5. 操作失败后显示错误提示。
6. 操作后返回当前详情页。
7. 中文 / English 文案都需要覆盖。

第一版只做单个标签添加 / 移除，不做多选标签管理。

## 六、本轮任务 5：返回列表时保留上下文

请优化从列表页进入详情页，再返回列表页的体验。

要求：

1. 从条目列表页点击条目详情时，应尽量带上当前列表 URL。
2. 详情页应提供“返回列表”按钮。
3. 返回列表时应保留：
   - 搜索关键词
   - 状态筛选
   - 标签筛选
   - 创作者筛选
   - 最低评分筛选
   - 时间范围
   - 排序
   - page
   - page_size

4. 可以使用 `next` 参数。
5. `next` 必须做安全校验。
6. 只允许站内相对路径。
7. 不允许跳转到外部 URL。
8. 不允许协议相对 URL，例如 `//evil.com`。
9. 如果 `next` 不安全或为空，应回退到 `/items`。
10. 不要因为异常 `next` 参数导致 500。

## 七、本轮任务 6：后端详情操作整理

请整理详情页相关后端逻辑。

建议新增 service：

- `app/services/item_detail.py`

也可以扩展现有 service，但不要让 router 变得过长。

要求：

1. 详情页编辑逻辑清晰可维护。
2. 对 item id / tag id / creator id 做基础校验。
3. 只操作本地 SQLite 数据。
4. 不拼接不安全 SQL。
5. 优先使用 SQLAlchemy ORM。
6. 对不存在的条目应返回 404 或合理错误。
7. 对不存在的标签 / 创作者应安全失败。
8. 对重复关联应避免重复创建。
9. 尽量使用事务。
10. 不引入外部任务队列。
11. 不引入后台异步任务。
12. 不引入新的依赖。
