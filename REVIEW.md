# Codex 产出审核 Checklist

每次 Codex 提交代码后，逐项检查：

## 超范围检测（一票否决）

- [ ] **0. Phase 1 中是否出现下列超范围内容：外部 HTTP 请求、爬虫、adapter、
  第三方 cookie/token 管理（自身登录 session 与语言偏好 session 除外）、远程图片拉取、自动同步、多源搜索、随机探索接口？**

> 出现任何上述内容 → 直接判定为**超范围**，退回 Codex 修复。这些属于 Phase 3。

## 基础检查

- [ ] **1. git status** — 是否只有本轮任务的变更文件
- [ ] **2. git diff** — 是否有预期之外的改动
- [ ] **3. 是否引入超范围功能** — 代码里是否有 Phase 2+ 的功能（推荐、AI、爬虫、外部源等）
- [ ] **4. 是否有硬编码路径** — 如 `/home/user/`、`C:\Users\` 等
- [ ] **5. 是否有敏感信息泄露** — API Key、密码、路径泄露
- [ ] **6. 是否有测试** — 对应功能的 `tests/` 文件
- [ ] **7. 类型注解** — 所有函数有类型提示
- [ ] **7.1. i18n 覆盖** — Web UI 中文 / English 文案是否完整，默认是否为中文
- [ ] **7.2. 页面提示体验** — 成功、错误、普通信息提示是否统一、可读且不泄露敏感信息

## 功能检查

- [ ] **8. API 是否符合路由设计** — 对照 PLAN.md 的 API 路由表
- [ ] **9. 数据库迁移是否安全** — 新增表/列不会破坏已有数据
- [ ] **10. 前端页面能否在局域网访问** — 不是 `localhost` 绑定
- [ ] **11. 错误处理** — 输入校验、404 处理、异常捕获
- [ ] **12. 日志是否脱敏** — 没有打印密码、cookie、token
- [ ] **12.1. 备份恢复是否安全** — 是否要求登录、只处理本地上传文件、不从 URL 恢复、失败时不破坏现有数据库

## Phase 2-A1 列表页增强检查

- [ ] **12.2. 高级筛选是否本地化** — 关键词、状态、单标签、单创作者、最低评分和时间范围是否只查询本地 SQLite 数据
- [ ] **12.3. 筛选参数是否稳定** — query string 是否可刷新、可复制，非法 `page` / `page_size` 是否安全回退且不触发 500
- [ ] **12.4. 排序是否完整** — 是否覆盖创建时间、更新时间、标题和评分的正反向排序，默认是否为最近更新
- [ ] **12.5. 分页体验是否完整** — 是否支持 `10` / `20` / `50` / `100` 分页大小，页码跳转是否保留筛选和排序条件
- [ ] **12.6. 列表页 UI 状态是否清晰** — 是否保留表单输入，展示当前筛选条件，提供清空筛选和无匹配结果提示
- [ ] **12.7. 列表页 i18n 是否完整** — 新增筛选、排序、分页和空状态文案是否同时覆盖中文 / English
- [ ] **12.8. Phase 2-A1 边界是否保持** — 是否未引入外部内容源、爬虫、adapter、远程图片拉取、自动同步、多源搜索、推荐系统或 AI 助手

## Phase 2-A2 批量编辑检查

- [ ] **12.9. 批量选择是否限于当前页** — 是否支持单项选择、全选当前页和取消选择，且未实现跨页全选
- [ ] **12.10. 批量操作是否要求登录** — 未登录执行批量操作是否跳转登录或被拒绝
- [ ] **12.11. 批量状态 / 标签 / 评分是否安全** — 非法状态、评分、缺失标签和不存在标签是否安全拒绝且不触发 500
- [ ] **12.12. 批量删除是否安全** — 是否有二次确认、危险提示，且只删除选中条目并清理本地关联数据
- [ ] **12.13. 批量操作结果是否清晰** — 成功 / 失败 flash 是否展示处理数、跳过数和明确原因
- [ ] **12.14. 批量操作后是否保留列表状态** — 是否通过安全站内 `next` 返回原筛选、排序、分页和 page_size
- [ ] **12.15. 批量编辑 i18n 是否完整** — 新增多选、批量状态、标签、评分、删除和错误提示是否同时覆盖中文 / English
- [ ] **12.16. Phase 2-A2 边界是否保持** — 是否未引入外部内容源、爬虫、adapter、远程图片拉取、自动同步、多源搜索、推荐系统、AI 助手或大型前端框架

## 登录保护检查

- [ ] **13. 密码是否从环境变量读取** — 不是硬编码
- [ ] **14. 是否禁止空密码**
- [ ] **15. 是否使用 session cookie**
- [ ] **16. `.env` 是否在 `.gitignore` 中**
- [ ] **17. 所有页面和 API 是否默认需登录**
- [ ] **17.1. 语言偏好是否刷新后保留** — `/set-language?lang=zh|en` 是否可用

## 部署检查

- [ ] **18. `docker compose build` 是否通过**
- [ ] **19. `docker compose up` 能否正常启动**
- [ ] **20. `/api/` 端点是否能 curl 通**
- [ ] **21. SQLite 文件是否生成在 `data/` 目录下**
- [ ] **22. Docker 卷挂载是否正确**
- [ ] **22.1. N100 / LAN 部署文档** — README 是否说明端口、`.env`、`./data` 持久化、安全边界和备份建议

## 发布前检查

- [ ] **22.2. README 版本状态** — 是否标明 `v0.1.0 / Phase 1 MVP`、功能清单、边界和已知限制
- [ ] **22.3. CHANGELOG** — 是否包含 `v0.1.0`、已完成内容、安全说明和已知限制
- [ ] **22.4. 版本号准备** — 已有版本号位置是否为 `0.1.0`；未自动创建 tag 或 GitHub Release
- [ ] **22.5. CI 简洁性** — 是否仍只安装开发依赖并运行完整测试，没有引入复杂发布流程
- [ ] **22.6. 文档 / CI 格式** — Markdown 文件是否为正常多行结构，CI workflow 是否为正常多行 YAML

## 备份恢复检查

- [ ] **23. JSON 导出是否完整** — 是否包含 items、tags、creators、item_tags、item_creators、user_item_states
- [ ] **24. CSV 导出是否可读** — 是否包含条目、标签、创作者、状态等本地字段
- [ ] **25. 恢复格式校验** — 非本项目导出的 JSON 是否拒绝
- [ ] **26. 恢复事务性** — 非法 JSON 或恢复失败是否不会破坏现有数据库
- [ ] **27. 本地边界** — 是否没有外部 URL 请求、自动同步或远程图片拉取
- [ ] **28. 备份预览** — 预览接口是否要求登录、只校验不写库、返回核心表数量和 schema
- [ ] **29. 上传大小限制** — 是否只限制备份恢复/预览上传，超限文件是否不会进入恢复逻辑
- [ ] **30. 错误提示** — 缺文件、非 JSON、超限、非法 JSON、schema 不匹配、缺字段和恢复异常是否有中英文友好提示

## 常见违规

```
❌ 引入外部爬虫代码（Phase 1 不做）
❌ 引入 requests/httpx 对外 HTTP 请求
❌ 写入第三方 token/cookie 管理逻辑（自身登录 session 与语言偏好 session 除外）
❌ 引入推荐/ML 依赖
❌ 修改 database.py 加入不需要的表
❌ 引入 JS 前端框架（用 HTMX 不是 React/Vue）
❌ 提交了 .env 文件（应在 gitignore）
❌ 密码硬编码在代码中（应从环境变量读取）
```

## 验收通过标准

全部 [✅] → 通知你确认
检查项 0 不通过 → 直接退回
其他 ≥1 个 [❌] → 退回 Codex 修复
